Bitácora

  • Mayo 2
    Actividad: Se investigaron las implicaciones legales del reconocimiento facial.
    Resultados: El proyecto cambió de enfoque debido a preocupaciones legales sobre privacidad, pasando de reconocimiento facial a un sistema auditivo para discapacitados visuales.
    Problemas: Riesgos legales y de privacidad identificados.
    Soluciones: Modificación del proyecto, descartando el reconocimiento facial.
  • Junio 14
    Actividad: Decisión de usar la cámara ESP32 para el reconocimiento de imágenes y un sistema de respuesta auditiva.
    Resultados: El sistema ayudaría a los usuarios a desplazarse con seguridad.
    Problemas: Complejidad en crear una base de datos para la ESP32.
    Próximos pasos: Investigar la ESP32 y evaluar alternativas si es necesario.
  • Junio 20
    Actividad: Presentación del proyecto sin mentor. Se detallaron los componentes clave (cámara ESP32 y sistema auditivo).
    Próximos pasos: Investigar más a fondo la ESP32 y realizar pruebas.
  • Junio 27
    Actividad: Reunión con mentor para evaluar la viabilidad técnica, materiales y costos.
    Resultados: Mejor comprensión de la magnitud del proyecto.
    Problemas: Dificultad para crear una base de datos para la ESP32.
    Soluciones: Investigar bases de datos públicas o cambiar la cámara.
  • Julio 12
    Actividad: Reunión con mentor sobre el sistema de audio.
    Resultados: Decisión de usar el DFPlayer Mini para la respuesta auditiva.
    Problemas: Falta de conocimiento sobre el DFPlayer Mini.
    Soluciones: Investigar su funcionamiento y compatibilidad.
  • Julio 13 y 14
    Actividad: Investigación de la cámara ESP32, enfocándose en el reconocimiento de imágenes y su precisión en diversas condiciones de luz.
  • Julio 17
    Actividad: Reunión con mentor. Se decidió explorar sensores ultrasónicos como alternativa a la ESP32.
    Resultados: Creación de un repositorio en GitHub y búsqueda de sensores ultrasónicos.
    Problemas: La ESP32 requiere mucho tiempo de calibración.
    Soluciones: Evaluar sensores ultrasónicos.
  • Julio 18
    Actividad Realizada: Decidimos utilizar el sensor ultrasónico HC-SR04 debido a su popularidad y facilidad de uso, lo que garantiza una amplia disponibilidad de recursos en línea para su implementación. Además, comenzamos a investigar sobre el DFPlayer Mini para integrarlo en el proyecto.
    Resultados: Encontramos varios tutoriales y recursos detallados sobre el funcionamiento del sensor ultrasónico HC-SR04, entre los cuales destacan:
    Watelectronics
    Robocraze
    YouTube
    Además, obtuvimos los esquemas y especificaciones de los sensores ultrasónicos que tenemos disponibles. También recopilamos información valiosa sobre el DFPlayer Mini de las siguientes fuentes:
    Maker.pro
    Electronics Lab
  • Julio 19
    Actividad Realizada: Desarrollamos un código que permite capturar fotos con la cámara ESP32 cada 2 segundos. Sin embargo, decidimos parar los avances relacionados con la cámara ESP32 para concentrarnos en la integración y desarrollo del sensor ultrasónico HC-SR04.
  • Julio 20
    Actividad Realizada: Con la ayuda de varios tutoriales y recursos sobre el funcionamiento del sensor, logramos obtener un código operativo para el sensor HC-SR04.
    Resultados: El sensor HC-SR04 está funcionando correctamente.
    Problemas: Es necesario adaptar el sistema para que funcione con múltiples sensores (cuatro en total).
    Próximos Pasos: Modificar el programa para que sea compatible con la integración de varios sensores HC-SR04.
  • Julio 21
    Actividad Realizada: Logramos adaptar el código originalmente diseñado para un sensor HC-SR04 para que funcione con tres sensores. Intentamos integrar el DFPlayer Mini para la reproducción de audio, pero el componente no está funcionando correctamente, lo cual podría deberse a problemas con la bocina utilizada.
  • Julio 24
    Actividad Realizada: Investigamos un código alternativo para el DFPlayer Mini que emplea una metodología diferente. En caso de que este nuevo código no funcione, consideraremos probar con una bocina diferente.
  • Julio 26
    Actividad Realizada: Recibimos nuevas bocinas y una batería que proporcionará la potencia necesaria para el funcionamiento óptimo del sistema.
    Próximos Pasos: Probar el programa del DFPlayer Mini con las nuevas bocinas y la batería.
  • Agosto 1
    Actividad Realizada: Se discutieron aspectos del sensor ultrasónico, la plataforma del prototipo y la altura óptima para maximizar su efectividad. Se exploraron otros sensores ultrasónicos, como el GY-US42V2.
    Resultados: Se estableció la plataforma y la altura para el montaje de los sensores ultrasónicos.
    Próximos Pasos: Investigar más sobre los sensores GY-US42V2 y realizar pruebas de montaje.
  • Agosto 9
    Actividad Realizada: Encontramos otros proyectos relacionados y recibimos los sensores GY-US42V2.
    Resultados: Se obtuvo una comprensión más detallada de los sensores, aunque con menos información disponible en comparación con el HC-SR04.
    Próximos Pasos: Integrar los GY-US42V2 en el prototipo y realizar pruebas iniciales.
  • Agosto 15
    Actividad Realizada: Continuamos investigando el sensor GY-US42V2 y consultamos recursos adicionales para obtener información sobre su funcionamiento.
    Próximos Pasos: Utilizar la información recopilada para desarrollar y programar el sensor GY-US42V2.
  • Agosto 19
    Actividad Realizada: Utilizamos un código diseñado para el GY-US42V2, pero el sensor no proporcionó lecturas correctas.
    Soluciones: Revisar y ajustar el código y las conexiones físicas del sensor.
    Próximos Pasos: Corregir el código y realizar pruebas diagnósticas.
  • Agosto 21
    Actividad Realizada: Identificamos que el sensor GY-US42V2 ofrece tres modos de operación: I2C, PWM y UART.
    Resultados: Comprensión mejorada de las capacidades del sensor y sus modos.
    Próximos Pasos: Evaluar cada modo de operación para determinar el más adecuado.
  • Agosto 22
    Actividad Realizada: Utilizamos el modo I2C del GY-US42V2, pero no logramos obtener resultados válidos.
    Resultados: El modo I2C no proporcionó lecturas útiles.
  • Agosto 23
    Actividad Realizada: Investigamos en línea para encontrar más información sobre el sensor GY-US42V2.
  • Agosto 25
    Actividad Realizada: Encontramos un código que inicialmente no funcionaba para el DFPlayer Mini, pero luego identificamos una versión que sí funcionó. Intentamos utilizar múltiples sensores GY-US42V2, pero esta configuración presentó problemas.
  • Agosto 27
    Actividad Realizada: Nos reunimos con un experto en robótica para recibir asesoría sobre las dificultades del proyecto.
    Resultados: Recomendaciones valiosas sobre problemas técnicos y estrategias de optimización.
    Próximos Pasos: Implementar las recomendaciones del experto y realizar pruebas.
  • Agosto 28
    Actividad Realizada: Logramos que el programa funcionara correctamente con dos sensores GY-US42V2 utilizando comunicación UART, gracias al soporte técnico de un experto.
  • Agosto 29
    Actividad Realizada: Tuvimos una segunda reunión en la que se discutió el avance obtenido tras la implementación del código funcional para el DFPlayer Mini y los sensores GY-US42V2. Se validó que el código está funcionando adecuadamente para las configuraciones actuales.
  • Septiembre 3
    Actividad Realizada: Desarrollamos un código para gestionar múltiples sensores GY-US42V2 en modo UART. Se realizaron pruebas para asegurar el funcionamiento correcto del sistema.
